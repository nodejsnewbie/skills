《Java高级编程》复习大纲：
一、设计模式的分类与理解
- 创建型模式
  - 单例模式
    - 实现方式（饿汉式、懒汉式）及其线程安全性。
    - 优缺点及适用场景。
  - 原型模式
    - 适用场景，如频繁创建对象且对象创建过程复杂。
  - 工厂方法模式
    - 定义、创建对象的职责分配。
    - 与抽象工厂模式的区别及使用场景。
  - 抽象工厂模式
    - 特点（创建一系列相关对象）。
    - 优点（易于交换产品系列、促进对象一致性）。
    - 缺点（难以支持新种类的产品）。
- 结构型模式
  - 适配器模式
    - 定义（将一个类的接口转换成客户希望的另一个接口）。
    - 适用场景（接口不兼容的类协同工作）。
  - 桥接模式
    - 作用（分离抽象与实现）。
    - 优缺点（增加设计难度，但降低耦合）。
  - 装饰模式
    - 核心目的（动态添加职责）。
    - 优点（不改变对象结构，动态扩展功能）。
    - 缺点（过度使用增加复杂性）。
  - 组合模式
    - 定义（将对象组合成树状结构）。
    - 优点（一致处理单个对象和组合对象）。
    - 两种实现方式（透明式、安全式）及其优缺点。
  - 外观模式
    - 类型（结构型模式）。
    - 优点（降低耦合度，减少客户端处理对象数目）。
    - 与适配器模式的区别。
- 行为型模式
  - 策略模式
    - 优点（支持开闭原则，灵活增加新算法）。
    - 适用场景（算法可变，需动态切换）。
  - 模板方法模式
    - 核心（定义算法骨架，延迟具体步骤到子类）。
    - 类型（行为型模式）。
  - 职责链模式
    - 优点（降低对象耦合度）。
    - 缺点（不能保证每个请求一定被处理）。
    - 应用场景（多个对象处理请求，动态决定处理者）。
  - 状态模式
    - 结构（环境角色、抽象状态角色、具体状态角色）。
    - 作用（封装与特定状态相关的行为）。
  - 观察者模式
    - 定义（描述对象间一对多依赖关系）。
    - 优点（支持广播通信，降低耦合）。
    - 缺点（可能导致依赖关系复杂）。
  - 中介者模式
    - 目的（解耦对象间交互，避免直接引用）。
    - 缺点（增加系统复杂性，违反单一职责原则）。
  - 迭代器模式
    - 定义（顺序访问聚合对象中元素）。
    - 关键角色（迭代器、聚合）。
    - 应用场景（遍历集合，隐藏内部结构）。
  - 访问者模式
    - 作用（不修改聚合元素前提下，添加新操作）。
    - 适用场景（对聚合对象操作复杂，需分离操作与对象结构）。
二、设计原则
- 开闭原则
  - 核心思想（对扩展开放，对修改关闭）。
- 单一职责原则
  - 定义（一个类只有一个职责）。
- 里氏替换原则
  - 定义（子类必须能够替换父类）。
- 依赖倒置原则
  - 定义（依赖抽象而不是具体实现）。
- 迪米特法则
  - 定义（降低类之间的耦合度，提高模块的相对独立性）。
- 接口分离原则
  - 定义（客户端不应依赖它不使用的接口）。
三、设计模式的应用场景
- 单例模式
  - 适用于需要全局唯一实例的场景，如配置管理器、线程池等。
- 工厂方法模式
  - 适用于创建多种产品，但具体产品类型在运行时确定的场景。
- 抽象工厂模式
  - 适用于创建一系列相关产品，且产品间有依赖关系的场景。
- 适配器模式
  - 适用于系统扩展，需要与旧系统接口兼容的场景。
- 装饰模式
  - 适用于需要动态增加对象功能，且不想使用继承的场景。
- 组合模式
  - 适用于对象间存在“部分 - 整体”关系，需要一致处理单个对象和组合对象的场景。
- 外观模式
  - 适用于简化复杂子系统接口，提供统一访问入口的场景。
- 策略模式
  - 适用于算法可变，需要动态切换算法的场景。
- 模板方法模式
  - 适用于算法骨架固定，但部分步骤需要子类实现的场景。
- 职责链模式
  - 适用于多个对象可能处理请求，动态决定处理者的场景。
- 状态模式
  - 适用于对象状态频繁变化，且状态相关行为不同的场景。
- 观察者模式
  - 适用于多个对象依赖于某个对象状态变化，需要广播通知的场景。
- 中介者模式
  - 适用于多个对象间交互复杂，需要解耦的场景。
- 迭代器模式
  - 适用于需要顺序访问集合元素，隐藏内部结构的场景。
- 访问者模式
  - 适用于对聚合对象操作复杂，需要分离操作与对象结构的场景。

五、设计模式的优缺点分析
- 单例模式
  - 优点：确保全局唯一实例，提供全局访问点。
  - 缺点：可能与多线程冲突，增加代码复杂性。
- 工厂方法模式
  - 优点：封装创建逻辑，易于扩展。
  - 缺点：增加类的数量，系统复杂度稍高。
- 抽象工厂模式
  - 优点：创建一系列相关对象，易于交换产品系列。
  - 缺点：难以支持新种类的产品，增加系统复杂性。
- 适配器模式
  - 优点：解决接口不兼容问题，提高复用性。
  - 缺点：增加类的数量，系统复杂度稍高。
- 桥接模式
  - 优点：分离抽象与实现，提高扩展性。
  - 缺点：增加设计难度，理解难度稍高。
- 装饰模式
  - 优点：动态添加职责，不改变对象结构。
  - 缺点：过度使用增加复杂性。
- 组合模式
  - 优点：一致处理单个对象和组合对象。
  - 缺点：客户端需要花更多时间理清类之间的层次关系。
- 外观模式
  - 优点：降低耦合度，减少客户端处理对象数目。
  - 缺点：不能很好地限制客户使用子系统类。
- 策略模式
  - 优点：支持开闭原则，灵活增加新算法。
  - 缺点：可能增加类的数量，系统复杂度稍高。
- 模板方法模式
  - 优点：定义算法骨架，易于扩展。
  - 缺点：可能限制子类的灵活性。
- 职责链模式
  - 优点：降低对象耦合度。
  - 缺点：不能保证每个请求一定被处理。
- 状态模式
  - 优点：封装与特定状态相关的行为。
  - 缺点：增加类的数量，系统复杂度稍高。
- 观察者模式
  - 优点：支持广播通信，降低耦合。
  - 缺点：可能导致依赖关系复杂。
- 中介者模式
  - 优点：解耦对象间交互，避免直接引用。
  - 缺点：增加系统复杂性，违反单一职责原则。
- 迭代器模式
  - 优点：顺序访问聚合对象中元素，隐藏内部结构。
  - 缺点：增加类的数量，系统复杂度稍高。
- 访问者模式
  - 优点：不修改聚合元素前提下，添加新操作。
  - 缺点：增加类的数量，系统复杂度稍高。
六、设计模式的综合应用
- 图形绘制系统
  - 使用工厂方法模式设计，实现不同图形类的创建。
- 员工请假审批系统
  - 使用职责链模式实现，根据请假天数动态决定审批者。
- 电子产品生产系统
  - 使用抽象工厂模式实现，创建不同品牌的产品。
- 音乐播放系统
  - 使用适配器模式实现，适配旧音频类以统一播放方法。
七、其他相关知识
- 设计模式的产生背景
  - 产生于建筑领域，由Christopher Alexander提出。
- 设计模式的主要目的
  - 提高代码的可复用性、可读性和可维护性。
- 设计模式对程序员能力的影响
  - 提高思维能力和编程能力，促进代码质量提升。
